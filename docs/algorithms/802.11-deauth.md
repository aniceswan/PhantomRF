# 802.11 deauth

> The "kick everyone off the WiFi" attack. Exploits a
> 25-year-old protocol design flaw.

802.11 management frames — the frames that control
association, authentication, and roaming — are
**unauthenticated**. A deauthentication frame can be forged
by any device on the channel, and the receiver has no way to
verify it came from a legitimate AP. This page is the deep
dive into the attack, the frame structure, and the
implementation.

For the user-facing documentation, see
[WiFi deauth](../features/wifi-deauth.md).

## The protocol design flaw

The original 802.11 standard, ratified in 1997, made a
choice that has come back to haunt every WiFi user since:
**management frames are not authenticated**. The thinking
was:

- The 2.4 GHz band was uncluttered; spoofing wasn't
  practical.
- Adding crypto to management frames would break
  compatibility with older devices.
- The attack surface (drive-by deauth) wasn't on anyone's
  radar.

In 2009, the standard was amended with **802.11w** (Protected
Management Frames, or PMF). PMF signs the management frames
with a key derived during the 4-way handshake. If the frame
isn't signed, the receiver rejects it.

As of 2026, **PMF is rarely enabled**. The WiFi Alliance
made it mandatory for WPA3 (2018+), but most networks still
run WPA2 with PMF disabled, because:

- Legacy devices (IoT, older phones) don't support PMF.
- Network admins don't realise it's an option.
- Some consumer routers ship with PMF off by default.

For a PhantomRF user, this means: 90%+ of the WiFi networks
you can see from the dashboard are vulnerable to deauth.

## The deauth frame structure

A 802.11 deauthentication frame is exactly 26 bytes. Here's
the structure, byte by byte:

```
Offset  Size  Field             Notes
------  ----  -----             -----
0       2     Frame Control     0xC0, 0x00  (Mgmt, Deauth)
2       2     Duration          0x00, 0x00
4       6     Destination MAC   The client to kick
10      6     Source MAC        The AP
16      6     BSSID             The AP, again (same as Source)
22      2     Sequence Control  0xF0, 0xFF
24      2     Reason Code       See table below
```

The Frame Control field `0xC0, 0x00` is composed of:

- Type: `00` (Management)
- Subtype: `12` (Deauthentication) → top byte `0xC0`
- Flags: `0x00` (none of: ToDS, FromDS, MoreFrag, Retry,
  PowerMgmt, MoreData, Protected, Order)

In little-endian, this is `0x00C0` stored as `0xC0, 0x00`.

The Destination MAC is the client to disconnect. The Source
MAC and BSSID are both the AP's MAC. The receiver doesn't
care that Source and BSSID are the same — it just checks
that Source matches the AP it's connected to.

> :information_source: **The BSSID is sometimes a virtual
> MAC**, different from the AP's hardware MAC. Modern
> multi-SSID routers (hostapd, UniFi, eero) often use a
> different BSSID per SSID. PhantomRF learns the BSSID by
> scanning first; if you see multiple BSSIDs for the same
> SSID, the firmware picks the one with the strongest
> signal.

## How to send raw 802.11 frames on the ESP32-S3

The ESP-IDF provides `esp_wifi_80211_tx()`, a function that
bypasses the normal WiFi stack and sends a raw 802.11 frame
at the current channel.

```c
esp_err_t esp_wifi_80211_tx(
    wifi_interface_t ifx,      // WIFI_IF_STA or WIFI_IF_AP
    const void*      buffer,   // pointer to the raw frame
    int              len,      // length of the frame
    bool             en_sys_seq // include the system sequence number
);
```

For deauth, the arguments are:

- `ifx` = `WIFI_IF_STA` (use the STA interface even in AP
  mode — the TX path is the same)
- `buffer` = the 26-byte deauth frame described above
- `len` = 26
- `en_sys_seq` = `false` (we control the sequence number)

Example call:

```cpp
DeauthFrame frame;
esp_wifi_80211_tx(WIFI_IF_STA, &frame, sizeof(frame), false);
```

The function returns `ESP_OK` on success, `ESP_FAIL` if the
hardware rejected the frame.

> :warning: **The function is rate-limited.** Espressif's
> firmware throttles raw 802.11 TX to about 10 frames/second
> (one per 100 ms). The first deauth usually disconnects the
> client; subsequent frames in the same 100 ms window are
> dropped. The PhantomRF firmware works around this by
> varying the sequence number on every frame so they're not
> deduplicated.

## The sniff-then-deauth trick

The most effective deauth is to **first sniff legitimate
frames from the target AP**, then **craft deauth using the
exact MAC addresses observed**. This avoids hard-coding
MACs and works for all clients associated with that AP.

The flow:

1. Set the WiFi radio to promiscuous mode.
2. Capture frames for 5–10 seconds.
3. For each frame, extract the BSSID, the source MAC, and
   any client MACs (from the Address fields).
4. Build a deauth frame for every (BSSID, client) pair.
5. Transmit the deauths.

The ESP32-S3's promiscuous mode callback receives every
frame on the channel:

```cpp
void sniffer(void* buf, wifi_promiscuous_pkt_type_t type) {
    const wifi_promiscuous_pkt_t* pkt =
        static_cast<wifi_promiscuous_pkt_t*>(buf);
    const uint8_t* mac = pkt->payload + 4;  // skip radiotap

    // Check that this is from an AP (Address2 == Address3)
    if (memcmp(mac, mac + 12, 6) == 0 &&
        memcmp(mac, "\xFF\xFF\xFF\xFF\xFF\xFF", 6) != 0) {
        memcpy(deauth.sta,  mac + 6,  6);  // client (DA)
        memcpy(deauth.ap,  mac,      6);  // AP (SA / BSSID)
        memcpy(deauth.src, mac,      6);  // source (we use AP's MAC)

        digitalWrite(2, HIGH);  // boost power (if wired)
        delay(15);              // 15 ms = "human-imperceptible"
        esp_wifi_80211_tx(WIFI_IF_STA, &deauth, sizeof(deauth), false);
        digitalWrite(2, LOW);
    }
}
```

The `15 ms` delay is a pattern from the W0rthlessS0ul
lineage: it makes the deauth look like a "normal" client
disconnect, which can fool some logging systems.

> :warning: **The "Address2 == Address3" check is what
> identifies a frame from an AP.** In 802.11, frames from
> an AP have the BSSID in both the Source Address (Address
> 2) and the BSSID field (Address 3). Frames from a client
> have different addresses.

## Reason codes

The 16-bit reason code tells the receiver *why* the
deauth was sent. Some receivers log the code; some use it
to decide whether to reconnect automatically. Common
values:

| Code | Name | When to use |
|---|---|---|
| 0x0001 | Unspecified | The default. Use this for "no good reason". |
| 0x0002 | Previous auth no longer valid | The client's old key expired. |
| 0x0003 | Deauth because sending STA is leaving | Client is moving out of range. |
| 0x0004 | Disassociated due to inactivity | Client hasn't sent data in N minutes. |
| 0x0005 | Disassociated because AP is unable to handle all currently associated STAs | AP is overloaded. |
| 0x0006 | Class 2 frame received from nonauthenticated STA | Client sent data before authenticating. |
| 0x0007 | Class 3 frame received from nonassociated STA | Client sent data before associating. |
| 0x0008 | Disassociated because sending STA is leaving BSS | Same as 3, but the client moved. |

PhantomRF defaults to 0x0001 (unspecified). You can change
it with `set wifi.deauth_method 0` (channel) → 1 (all
channels) → 2 (smart sniff) in the CLI.

## PMF (802.11w) and what it does

PMF (Protected Management Frames) is the 802.11w amendment
that signs the management frames. With PMF enabled:

- Every deauth frame is signed with a key derived from the
  4-way handshake.
- The receiver checks the signature before acting on the
  frame.
- Unsigned deauth frames are silently discarded.

PMF is mandatory for **WPA3** (2018+) and optional for WPA2.
It is sometimes called "Management Frame Protection" or
"802.11w" interchangeably.

### How to tell if a network has PMF

From a WiFi scan, you can't tell directly. The RSN
information element in the beacon includes a "MFPC" bit
(Management Frame Protection Capable) and an "MFPR" bit
(Management Frame Protection Required). PhantomRF
displays these in the scan results as "PMF: optional" or
"PMF: required".

When PMF is **required**, the deauth attack will not work
against clients that respect the standard. Modern iPhones
and Macs do respect it.

When PMF is **optional** (or not present), the deauth
attack works against all clients on the network.

### Defeating PMF

There is no public method to defeat PMF on a properly
configured WPA3 network. The cryptographic key is derived
from the 4-way handshake and never transmitted in the clear.
A successful attack would require:

- A weakness in the cipher (none known for CCMP/GCMP)
- A side-channel (timing, power) — research only
- Physical access to the AP or client

PhantomRF does not attempt any of these. The deauth attack
is a standards-compliance test, not a cryptographic attack.

## Limitations

- **PMF defeats the attack.** See above. As more networks
  adopt WPA3, this attack will become less effective.
- **The ESP32-S3's raw TX is rate-limited.** Espressif
  throttles to ≈ 10 frames/second. After the first deauth,
  the client is gone, and subsequent frames are wasted
  (unless the client tries to reconnect).
- **No effect on a wired network.** Ethernet clients are
  unaffected.
- **The PhantomRF AP itself is a 2.4 GHz device.** A
  deauth on a real network will also disrupt the PhantomRF
  AP's link. See
  [DESIGN §20.1](https://github.com/anomalyco/PhantomRF/blob/main/DESIGN.md#201-mitigation-self-jamming--rf-co-existence).
- **Detection is possible.** Modern APs and WIDS (Wireless
  Intrusion Detection Systems) can spot deauth floods by
  counting the rate of deauth frames. PhantomRF does not
  try to hide.

## C++ implementation notes

The PhantomRF code wraps the deauth in a `WifiDeauth` class
in `src/modules/wifi/WifiDeauth.cpp`:

```cpp
class WifiDeauth {
public:
    void begin();
    void setTarget(const MacAddress& bssid);
    void start();
    void stop();
    void setReasonCode(uint16_t code);
private:
    MacAddress     bssid_{};
    bool           running_ = false;
    TaskHandle_t   task_    = nullptr;
    void deauthTask();
};
```

The deauth task runs on Core 0 (the same core as the WiFi
stack). It maintains a small ring buffer of recent client
MACs seen in promiscuous mode and deauths each one in turn.

The promiscuous-mode callback is registered with:

```cpp
esp_wifi_set_promiscuous(true);
esp_wifi_set_promiscuous_rx_cb(&sniffer);
```

The callback receives every frame on the current channel.
We filter for management frames from APs (Address 2 ==
Address 3, type = 0) and extract the client MACs.
